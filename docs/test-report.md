## 3. 并发一致性场景

测试创建 30 个互不相同的用户，同时抢购库存为 10 的同一活动。每个线程使用独立 MySQL 连接，并通过 `std::barrier` 同时开始请求。

| 指标 | 结果 |
| --- | --- |
| 并发用户 | 30 |
| 初始库存 | 10 |
| 抢购成功 | 10 |
| 售罄或活动不可用 | 20 |
| 最终库存 | 0 |
| 新增订单 | 10 |
| 新增库存流水 | 10 |
| 重复用户订单组 | 0 |

测试还会让一个成功用户再次抢购，必须返回 `DuplicateOrder`。

## 4. 一致性结论

- 条件更新 `available_stock > 0` 保证库存不会扣成负数。
- 唯一索引 `(activity_id, user_id)` 保证同一用户不能重复下单。
- 库存扣减、订单创建和库存流水写入位于同一事务。
- 成功路径显式提交；异常路径由 RAII `Transaction` 析构回滚。
- 并发测试结束后校验 `最终库存 = 0` 且 `订单数 = 流水数 = 初始库存`。

## 5. 当前限制

- 当前已包含 epoll HTTP 服务和原生 Web 前端，但未实现 io_uring、业务线程池、连接池和 Vue 工程化。
- 当前密码仅使用 SHA-256 演示，不适用于生产系统。
- Web 登录会话保存在进程内存中，服务重启后需要重新登录。
- 并发测试用于验证一致性，不等同于完整吞吐量压测；后续接入网络层后再统计 QPS 和延迟分位数。
